01 — Foundation

Why Structure Matters

Structure is the set of rules that defines where things live and who may touch them. Without it, every growing codebase eventually becomes an archaeology project.

🔍

Discoverability

A new developer should be able to find any feature file within 60 seconds. Clear naming conventions and predictable folder placement make orientation trivial — not an onboarding challenge.

🧩

Module Boundaries

Each feature is self-contained. A change to the cart feature shouldn't break auth. Clear boundaries prevent invisible coupling.

📈

Scalability

Structure preserves order as the project grows. What works at 5 files can collapse at 500. Planning the architecture early is far cheaper than refactoring later under deadline pressure.

The problems without structure

When any file can import any other file, you create invisible dependencies. Changing a utility in shared/ can crash five unrelated features. Without enforced import direction, the dependency graph becomes a plate of spaghetti.


Cart → Auth → Utils → Cart ❌

Circular dependency: Cart → Auth → Utils → Cart. This breaks at build time or causes subtle runtime bugs.

In a flat components/ directory, you eventually get Button.tsx, ButtonNew.tsx, ButtonV2.tsx. No one knows which is canonical. Feature folders solve this: the button inside features/auth/ui/ is unambiguously the auth-specific button.

Without colocation, deleting a feature leaves orphan files scattered across the tree — CSS files with no component, hooks no one calls, API functions long-superseded. Colocation means deleting a folder deletes everything that belongs to it.

When different developers interpret "where does this go?" differently, PRs become debates. A documented, agreed-upon architecture is a shared language. It's also the difference between 10-minute onboarding and 10-day onboarding.

Three goals of good structure

  • Readability — a developer unfamiliar with the codebase can find any feature file in under a minute. Folder names match the business domain, not technical abstractions.
  • Module boundaries — each feature encapsulates its own UI, logic, and API calls. The public API (what it exports) is small and intentional.
  • Scalability — the system should still make sense at 10× the current file count. Structure is a growth multiplier, not a constraint.
Senior tip: "This decision framework is the map. The territory (Router, React Hook Form, Context) is covered separately — the architecture layer is where everything connects."
02 — Patterns

Architecture Patterns

Different scales and teams call for different patterns. Understanding the trade-offs is more important than picking "the right one."

Flat component structure

The traditional approach: every component lives in one components/ directory. Fine for prototypes, painful at scale.

├─src/
│ ├─components/
│ │ ├─Button.tsx
│ │ ├─LoginForm.tsx
│ │ ├─CartDrawer.tsx
│ │ ├─ProductCard.tsx
│ │ └─... (eventually 80+ files)
│ ├─App.tsx
│ └─index.tsx

Verdict

AspectRating
Setup speed✓ Fastest
Discoverability (small)✓ Good
Discoverability (large)✗ Terrible
Team scalability✗ Poor
Refactor cost~ Medium
Use flat structure for: demos, side projects under ~20 components, or a single-developer prototype you'll refactor before shipping.

Feature folder structure

Each business domain gets its own folder containing UI, API calls, and hooks. The feature is the unit of organisation.

├─src/
│ ├─features/
│ │ ├─auth/
│ │ │ ├─ui/ LoginForm.tsx
│ │ │ ├─api/ login.ts
│ │ │ └─model/ useAuth.ts
│ │ ├─cart/
│ │ │ ├─ui/ CartDrawer.tsx
│ │ │ ├─api/ cartApi.ts
│ │ │ └─model/ useCart.ts
│ │ └─checkout/ ...
│ └─shared/ ui/ lib/ hooks/

Key principles

  • Each feature imports only from shared/ — never from another feature directly.
  • The index.ts barrel in each feature defines the public API — internal files stay private.
  • Deleting a feature means deleting one folder, with zero orphan files.
  • Teams can "own" features — reducing merge conflicts and cognitive load.
Best for: Teams of 2–10+ developers shipping a real product. The most common production pattern in 2024.

Layered / Slice architecture

Extends feature-based with explicit layers: app, pages, widgets, features, entities, shared. Each layer may only import from layers below it.

app/ ← root, providers, global router
↓
pages/ ← URL = composition layer
↓
widgets/ ← large UI blocks (DashboardShell)
↓
features/ ← business logic flows
↓
entities/ ← product, user (single domain)
↓
shared/ ← primitives, api client, constants

The import rule (enforced)

Each layer only imports from layers below it. Same-level slices may NOT cross-import — orchestration happens in the layer above.

// .eslintrc.js — enforce the rule rules: { "import/no-restricted-paths": ["error", { zones: [ { // features cannot import from pages target: "./src/features", from: "./src/pages" }, { // shared cannot import from features target: "./src/shared", from: "./src/features" } ] }] }

Feature-Sliced Design (FSD)

An open standard for large-scale React applications. Every slice has a public API (barrel), and same-layer slices never touch each other.

├─src/
│ ├─app/ # Init, providers, global router
│ ├─pages/ # Route = composition only
│ ├─widgets/ # Complex self-sufficient UI blocks
│ ├─features/ # Business processes
│ │ └─auth/
│ │ ├─index.ts # Public API ← IMPORTANT
│ │ ├─ui/
│ │ ├─model/
│ │ ├─api/
│ │ └─lib/
│ ├─entities/ # Domain models (product, user)
│ └─shared/
│ ├─ui/ lib/ api/ config/

When to use FSD

  • Large teams (5+ developers) on a long-lived product.
  • Multiple business domains that need clear ownership.
  • You need to enforce import rules via linting (not just convention).
  • The product will grow — you're building infrastructure, not a prototype.
Important: FSD adds overhead. For a small team or MVP, start with feature folders and evolve. The official spec is at feature-sliced.design.
✓ Enforced by lint ✓ Public APIs ⚠ High overhead
03 — Colocation

Keep related files close

Colocation means the test, style, and hook for a component all live in the same directory. Change one thing, find everything in one place.

Without colocation

├─components/ ProductCard.tsx
├─styles/ ProductCard.module.css
├─hooks/ useProductCard.ts
└─tests/ ProductCard.test.tsx

⚠ Four files in four places. Delete the component → three orphans remain.

With colocation ✓

└─ProductCard/
├─ProductCard.tsx
├─ProductCard.test.tsx
├─ProductCard.module.css
└─useProductCard.ts

✓ One folder = one concern. Delete folder → everything gone. IDE navigation instant.

Barrel files — index.ts

A barrel re-exports everything the module wants to expose, giving external consumers a clean import path.

// features/auth/index.ts (barrel) export { LoginForm } from './ui/LoginForm'; export { useAuth } from './model/useAuth'; export type { AuthUser } from './model/types'; // ✗ NOT exported — stays private: // ./api/refreshToken.ts // ./ui/OAuthButton.tsx
// Consumer — clean import import { LoginForm, useAuth } from '@/features/auth'; // NOT: import LoginForm from '@/features/auth/ui/LoginForm'

Barrel trade-offs

✅
Clean imports Consumers never need to know internal structure. Move files within the feature without breaking consumers.
✅
Explicit public API If it's not in the barrel, it's private. This is enforced by convention (and can be enforced by lint).
⚠️
Circular dependency risk A → B (via barrel) → A creates a circular import. Always import from below in the layer hierarchy.
⚠️
Tree-shaking in some bundlers Over-exporting from barrels can confuse some bundlers. Keep barrels shallow and avoid re-exporting entire third-party libraries.
04 — Reference

Scalable project structure

The reference tree for a large React application using layered architecture. Every folder has a single clear responsibility.

src/
├─app/ # App shell
│ ├─router.tsx # Route definitions
│ ├─providers.tsx # Global context wrappers
│ └─App.tsx
│
├─pages/ # One file per route
│ ├─HomePage.tsx
│ ├─ProductPage.tsx
│ └─CheckoutPage.tsx
│
├─features/ # Business domain
│ ├─auth/
│ │ ├─index.ts # Public API
│ │ ├─ui/ LoginForm.tsx
│ │ ├─api/ authApi.ts
│ │ └─model/ useAuth.ts
│ ├─cart/ ...
│ └─checkout/ ...
│
├─entities/ # Domain models
│ ├─product/ ProductCard.tsx, types.ts
│ └─user/ UserAvatar.tsx, types.ts
│
└─shared/ # No domain knowledge
├─ui/ Button, Input, Modal ...
├─hooks/ useDebounce, useLocalStorage
├─api/ httpClient.ts
├─lib/ formatters, validators
└─config/ constants, env

Layer responsibilities

app/

Root layout, router, global state providers. No business logic. This wires everything together.

pages/

One file per URL. Pages are pure composition — they pull in features and widgets and arrange them. No data fetching logic here.

features/

Business processes: auth, cart, checkout. Each feature owns its own UI, API, and state logic. The most important layer.

shared/

Project-agnostic primitives. A Button component here knows nothing about products or users. If it does, it's in the wrong place.

Decision rule: Ask "does this know about a specific business domain?" — if yes, it belongs in features/ or entities/. If no, shared/.
05 — Process

UI planning before you code

Great architecture starts on paper (or a whiteboard). Code before planning is like building without blueprints — fast to start, expensive to fix.

The 3-step UI planning method

1

Screen map + data sources

List every screen with its URL. For each screen, identify the primary data source — which component fetches the data and from where. This prevents "data waterfalls" later.

2

Break down: layout → section → atom

Start with the outermost container (shell, sidebar, header). Then identify logical sections (filter panel, product list, detail pane). Finally, identify repeating primitives that belong in shared/ui/.

3

Plan all UI states

For every data-driven UI, define four states: loading, empty, error, success. Skipping these means TODO comments that never get resolved.

State planning

Before writing a single useState, categorise every piece of state.

State Type Lives In Tool
Server data Query cache TanStack Query
Global UI Store Zustand / Context
Form state Form library React Hook Form
Local UI Component useState
URL state Address bar useSearchParams
The most common mistake: putting server data into useState + useEffect when React Query would handle caching, background refetching, and error states for free.

Wireframe thinking + user flow

A wireframe doesn't need to be pretty. It needs to answer: where is the nav, where is the content, where are the CTAs. Structure first, pixels later.

            ┌─────────────────────────────────────┐
            │ NAVBAR      [Logo]   [Nav]  [CTA]   │
            ├────────────┬────────────────────────┤
            │            │                        │
            │ SIDEBAR    │    PRODUCT LIST        │
            │            │   ┌───┐ ┌───┐ ┌───┐    │
            │ Filters    │   │   │ │   │ │   │    │
            │ Categories │   └───┘ └───┘ └───┘    │
            │ Price      │   ┌───┐ ┌───┐ ┌───┐    │
            │            │   │   │ │   │ │   │    │
            │            │   └───┘ └───┘ └───┘    │
            ├────────────┴────────────────────────┤
            │              PAGINATION             │
            └─────────────────────────────────────┘

A user flow maps each decision point and screen transition. Translating this to React means mapping flows to route changes and UI state transitions.

06 — Components

Component strategy

Not all components are created equal. Separating "smart" data containers from "dumb" presentational components is the fastest route to reusability and testability.

🗺️

Page / Route Component

Tied to a URL. Its job is composition only — it assembles features and shared components. It may trigger top-level data loading (via loader or query) but delegates everything else.


export default function ProductPage() { return ( <Shell> <FilterBar /> <ProductGrid /> </Shell> ); }
🧠

Container (Smart)

Handles data fetching, side effects, and state. Passes results down as props. Usually not directly reusable — it's purpose-built for one feature.


function ProductGridContainer() { const { data, isLoading } = useProducts(); if (isLoading) return <Skeleton />; return <ProductGrid items={data} />; }
🎨

Presentational (Dumb)

Takes props, renders UI. No side effects, no fetching. This is the component you put in Storybook, the one that's easy to test, the one that can be reused anywhere.


function ProductGrid({ items }: { items: Product[] }) { return ( <Grid> {items.map(i => <ProductCard key={i.id} {...i}/>)} </Grid> ); }

The "rule of three" for abstraction

❌
Premature abstraction You see a component used twice, immediately move it to shared/ and add 12 props for flexibility. Now both consumers are shaped around the abstraction instead of their own needs. Wait.
❌
"Uber-prop" components A component with 20 props to handle every edge case is not reusable — it's a micro-framework. If you need that many props, consider composition patterns instead.
✅
Duplicate first, then extract Copy the component twice. On the third usage, the shared shape becomes obvious. Extract then. This is the "rule of three."
✅
Composition over configuration Instead of a Modal with 15 props, export Modal.Header, Modal.Body, Modal.Footer. Consumers assemble what they need.
07 — Ecosystem

React ecosystem map

The ecosystem is vast. This is the curated list of packages a senior developer reaches for — and why each one earns its place.

🧭

react-router

Routing

Maps URLs to UI components. Supports nested routes, dynamic segments (:productId), and data loaders that fetch before rendering. The browser's back button works correctly because history is managed properly.

const router = createBrowserRouter([ { path: "/", element: <Shell />, children: [ { path: "products/:id", element: <ProductPage /> } ] } ]);
⚡

TanStack Query

Server State

Manages all server-sourced data: caching, background refetching, stale-while-revalidate, and error states. Eliminates the useEffect + useState data-fetching pattern that litters most codebases.

const { data, isLoading, error } = useQuery({ queryKey: ['products', filters], queryFn: () => fetchProducts(filters), staleTime: 5 * 60 * 1000, // 5 min cache });
📝

React Hook Form

Forms

Handles complex forms with minimal re-renders. Uses uncontrolled inputs by default, integrates with Zod/Yup for schema validation, and binds errors directly to fields. Replace 200 lines of hand-rolled form logic with 20.

const { register, handleSubmit, formState } = useForm({ resolver: zodResolver(schema) }); <input {...register("email")} /> <p>{formState.errors.email?.message}</p>
🐻

Zustand

Client State

Minimal global state store. Avoids Context's re-render avalanches. Use it for UI-only global state: theme, sidebar open/closed, shopping cart, notifications. Server state belongs in TanStack Query — not here.

const useCartStore = create<CartStore>()(set => ({ items: [], add: (item) => set(s => ({ items: [...s.items, item] })) }));
🎨

shadcn/ui + Radix

UI Components

Not an npm package — you copy components into your repo. Built on accessible Radix primitives + Tailwind. You own the code: customise freely without fighting library constraints. The new default for greenfield projects.

  • Full accessibility (ARIA, keyboard nav, focus trap)
  • Your code — no version lock-in
  • Tailwind-based, easy to restyle
🧪

Vitest + Testing Library

Testing

Vitest uses the same Vite config as your app — no Jest migration pain. Testing Library tests the component from the user's perspective: screen.getByRole over querySelector. Tests survive refactoring because they test behaviour, not implementation.

test('submits login form', async () => { render(<LoginForm />); await userEvent.type( screen.getByLabelText(/email/i), 'a@b.com' ); await userEvent.click(screen.getByRole('button')); expect(mockLogin).toHaveBeenCalled(); });
08 — Best Practices

What senior devs do differently

Architecture is only as good as the habits that maintain it. These are the patterns that separate a codebase that ages well from one that becomes legacy on day 60.

⚡ Performance

Code splitting at the route level

Lazy-load page components so the initial bundle only contains what the user sees first.

const CheckoutPage = lazy( () => import('./pages/CheckoutPage') ); // React Router 6 data router does this // automatically with lazy() option

Memoisation — use sparingly

useMemo and useCallback have a cost (memory, complexity). Profile first, optimise second. The most effective optimisation is avoiding unnecessary renders by splitting components at the right boundaries.

  • Use memo() only when profiling shows a real problem
  • TanStack Query's built-in caching is free performance

🚫 Anti-patterns to avoid

❌
Everything in App.tsx One 800-line file that is router, state, API fetcher, and layout. Untestable, unreadable, unmergeable.
❌
Prop drilling 5 levels deep Passing data through 5 components that don't use it. Use Context, Zustand, or component composition to lift state correctly.
❌
Server state in useEffect + useState useEffect(() => { fetch(...).then(setData) }, []) everywhere. This is exactly what TanStack Query replaces — with caching, loading states, and deduplication.
❌
Skipping the "empty" and "error" UI states Planning only the "success" state leaves your app looking broken to real users. Design all four states from the start.
